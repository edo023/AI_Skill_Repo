# Operational Limitations of AI-Assisted Engineering Systems

Artificial Intelligence (AI)-assisted engineering systems, including generative and retrieval-based models, are increasingly used to support engineering workflows. However, these systems have fundamental limitations that can lead to errors, logical flaws, or unsafe outputs — even when relevant data exists in the training data or knowledge base.

This AI Assistant is intended solely as a productivity tool to support experienced automation engineers. It does not replace professional engineering judgment, formal design reviews, or mandatory verification and validation processes required for industrial control systems.

AI-generated code may contain subtle logical errors, race conditions, incorrect timer/counter behavior, unsafe state transitions, or improper handling of fault conditions that are not obvious on casual inspection.

All outputs generated by the assistant, particularly those involving automation or control code, must be reviewed and validated through engineering quality assurance procedures before use in production environments.

Your feedback is actively used to improve the accuracy, contextual understanding, and overall reliability of the assistant over time. However, user feedback and iterative improvements do not eliminate the fundamental limitations described below, nor do they remove the requirement for full engineering validation of every output.

1. **Variability in Responses (Non-Deterministic Behavior)**

   * **Description**: Identical prompts may produce different outputs across sessions, including variations in generated control logic.
   * **Cause**: The assistant relies on probabilistic generation, which can yield multiple, seemingly valid but behaviorally different interpretations.
   * **Controller-specific risk:** Even small differences in code structure, condition ordering, or error handling can lead to inconsistent machine behavior or unintended safety gaps.
   * **Best Practice**: For repeatable or safety-critical tasks, use highly structured and constrained prompts. Generate and compare multiple versions, then document and version-control only the validated output.
2. **Factual Inaccuracies and Logical Errors in Code**

   * **Description**: The assistant may generate inaccurate incomplete, or logically inconsistent information and code, even when correct data exists in the knowledge base or training material.
   * **Cause**: Generative models predict plausible responses rather than performing true engineering reasoning or formal verification. They can misinterpret controller-specific instructions, timing behavior, or safety requirements.
   * **Best Practice**: Always validate outputs against the latest official Schneider Electric documentation and the established standards and coding guidelines of your organization. Use precise, context-rich prompts to help improve reliability. Never rely on the AI-Assistant for safety-related logic.
3. **Incomplete, Incompatible, or Non-Optimal Code**
   * **Description**: Code generated or retrieved may be incomplete or incompatible with your specific Schneider Electric system configuration, or transgress hardware constraints (memory limits, scan cycle time, I/O addressing, etc.).
   * **Cause**: Limited context or partial matches in the retrieval process.
   * **Best Practice**: Always include system-specific information in your prompts (PLC reference, firmware version, relevant hardware configuration, etc.). Manually review, complete, adapt, and test all generated code in the Schneider Electric development environment (e.g., EcoStruxure Machine Expert). Perform compilation and resource usage verifications before any deployment.
4. **Outdated or Irrelevant Retrievals**

   * **Description**: The assistant may retrieve or generate content based on legacy documents, deprecated instructions, or practices that no longer comply with current standards.
   * **Cause**: The knowledge base includes historical data that may not reflect present-day standards.
   * **Best Practice**: Always cross-check against the most recent official Schneider Electric documentation and applicable standards. Report any outdated content via feedback for review and updating.
5. **Conflicting Information Across Sources**

   * **Description**: Responses may include contradictory statements or code examples.
   * **Cause**: The assistant integrates content from multiple Schneider Electric sources, which may reflect differing practices or legacy standards.
   * **Best Practice**: Use the most recent Schneider Electric documentation. Consult your engineering guidelines when discrepancies arise.
6. **Hallucinated or False Safety-Critical Behavior**

   * **Description:** The assistant may generate code that appears to implement safety functions (e.g., emergency stop handling, guard monitoring, safe torque off) but contains hidden logical flaws.
   * **Cause:** The model lacks the ability to perform formal safety analysis, reference verification, or exhaustive fault scenario evaluation.
   * **Controller-specific risk:** A false sense of security; the code may pass basic tests but fail under real fault conditions or specific timing scenarios.
   * **Best Practice:** Never use AI-generated code for any safety-related functions. All safety logic must be developed, verified, and validated exclusively by qualified safety engineers using certified tools and methodologies.
7. **Lack of Formal Verification and Certification**

   * **Description:** AI systems cannot provide mathematically proven correctness, exhaustive state-space analysis, or compliance with functional safety certification requirements.
   * **Best Practice:** For applications requiring Safety Integrity Level (SIL) 2 or higher, or Performance Level (PL) d/e, rely solely on traditional engineering processes, certified safety controllers, and validated function blocks. The AI Assistant and its outputs are not certified for use in safety-related systems.

## Conclusion and User Responsibility

The AI Assistant is designed as a supportive productivity tool to assist experienced automation engineers in generating ideas, code snippets, and explanations for Schneider Electric PLC applications. Despite continuous improvements through user feedback, it remains subject to the inherent limitations outlined above. Users bear sole responsibility for the review, adaptation, verification, validation, and safe deployment of any AI-generated content.

All outputs — especially control logic, Program Organization Units (POUs), and safety-critical code — must be rigorously tested and validated using established engineering processes, the latest official Schneider Electric documentation, and applicable safety standards before use in any machine or process.
