# Program Organization Units (POUs)

## `SR_EmailHandling`

The program `SR_EmailHandling` implements the program code to control the email client (for sending and receiving emails) and the processing of received emails.

The program code has been split into single logical functions. The single program parts are processed in subroutines (actions).

The actions called from the program are listed in the following table:

| Name of the subroutine (action) | Description |
| --- | --- |
| `ACT_InputMapping` | In this action, the input variables, mainly coming from the visualization, are verified and assigned to local variables for further processing. |
| `ACT_CheckForSendRequest` | In this action, the commands to send an email are stored in a queue for sequential processing. |
| `ACT_UpdateCredentials` | In this action, the credentials and the communication settings for the email client are updated with the variables displayed on the visualization. Further functions such as, the hidden credentials are processed in this subroutine too. |
| `ACT_StateMachine` | This action implements the state machine for the email handling. The general states are:   * Initializing the function blocks * Monitoring the control commands * Triggering the command on the function blocks * Monitoring the execution of the function blocks |
| `ACT_AutoProcessing` | The auto processing is executed in a chronological interval and controls the email client as follows:   * Verify whether new emails are in inbox * Download new emails * Process the received emails: emails are deleted or a response is created and sent |
| `ACT_RearrangeQueueEmailToSend` | This action is called after each successful sending. It rearranges the queue for the buffered send requests. |
| `ACT_OutputMapping` | In this action, the local variables are converted into the required format and then assigned to the associated output variables. The output variables are linked to the visualization. |

## `SR_MachineData`

The program `SR_MachineData` implements a routine to create the content for emails which are sent on dedicated events. The example application implements the events MachineErrorState and DailyStatus which can be triggered from the visualization.
