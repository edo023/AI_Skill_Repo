# Operating the Application

## Visualization

The application example implements a visualization for monitoring and control purposes.

This section provides a brief description of each function. In addition, refer to the [visualization screen](D-SE-0077434.html#D-SE-0077434).

## Configuration

Before you can communicate with the email server, you need to configure the communication settings and the credentials for authentication. In the visualization screen Settings, you can modify the required configuration parameters.

In case of TLS encryption, accepting only trusted certificates, the server certificate has to be known and rated as trusted. For more information about certificate management on the controller, refer to [*How to Manage Certificates on the Controller - User Guide*](../../../../../api/crossBook?lang=en-US&virtualBookName=HowMgCer&topicID=D_SE_0095876).

In addition, other significant application parameters can be configured from the visualization screen. These include:

* Default email addresses for the recipients of emails generated on dedicated events
* Allowlist for filtering the emails to be downloaded
* Directory for saving downloaded email attachments

## Send an Email

Sending an email from the controller can be done from the visualization screen Send email. Enter one or more recipients, the subject, and the content of the email, the file path for the attachment, the priority and then click the button Send email.

The status of the function block `FB_SendEmail` is displayed in the lower area of the visualization screen, where you can monitor the status of the sending request.

In addition, you have the possibility to trigger events which causes an automatic creation and sending of emails. Clicking the respective button, the email is generated and sent to the configured recipient.

## Receive Emails

Reception of emails can be controlled from the visualization screen Receive email. The reception is managed by `FB_Pop3EmailClient` and provides three control commands which can be triggered by the respective buttons from the visualization.

| Command | Function |
| --- | --- |
| CheckInbox | Requests the function block to connect to the server and to verify whether there are new unread emails. If the execution of the function block is completed successfully, the number of downloadable emails on the server is displayed. |
| Receive | Requests the function block to download the selected emails. If the execution of the function block is completed successfully, the received emails are listed in the table on the visualization screen. |
| Delete | Requests the function block to delete the selected emails from the server. If the execution of the function block is completed successfully, the number of deleted emails is displayed. |

The status of the function block `FB_Pop3EmailClient` is displayed in the lower area of the visualization screen, where you can monitor the status of the requested command.

The table on the visualization screen lists the received emails. Emails with status as Ok have been downloaded to the controller and deleted from the server. Emails with a different status have been left on the server as unread email. For the selected email in the list, the details are displayed in the associated fields in the visualization screen.

## Auto Processing

The application example implements a subroutine for automatic processing.

The auto processing is executed in a chronological interval and controls the email client as follows:

* Verification whether new emails are in inbox
* Downloading new emails
* Processing of the received emails: emails are deleted or a response is created and sent

The auto processing can be activated from the visualization screen. The interval for executing the auto processing can be configured. The control mask for the auto processing is displayed on each visualization screen.
