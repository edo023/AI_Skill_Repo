# Multi-Axis Custom Jobs

## Overview

The library MultiAxisCustomJob provides a framework for coordinating the movements of multiple subordinate axes in each RTP cycle using a centralized algorithm. This algorithm can also reference the positions of multiple master axes.

This library is intended for complex mechanical systems where axes are kinematically coupled, such as robotic articulations, where the movement of one axis directly affects the required position of others.

To develop your motion logic, you must create a function block that implements the interface MACJ.IF\_MultiAxisCustomJobCallback.

The library consists of three primary function blocks:

* FB\_MultiAxisCustomJob: The central function block for the coordination and management of the axis group.
* FB\_SlaveChannel: The interface for each subordinate channel to be controlled by the custom algorithm.
* FB\_MasterChannel: The interface for each master channel providing reference values for the calculation.

## Setting up a Multi-Axis Custom Job

FB\_MultiAxisCustomJob  is the central function block used to manage the axis group. To define the group, you must connect the individual channel function blocks to the administration block before starting the job.

The group is set up by calling specific methods to link channel function blocks with their physical or virtual axes:

* Subordinate channels: For every axis to be controlled by the user-defined algorithm, you must create an instance of FB\_SlaveChannel. Connect this channel to the group by calling the method AddSlaveChannel of the function block FB\_MultiAxisCustomJob, providing the channel instance and the corresponding axis as inputs.
* Master channels: If the calculation requires reference values from master axes, create an instance of FB\_MasterChannel for each. Connect these to the group by calling the method AddMasterChannel, providing the channel instance and the reference axis as inputs.

You must connect at least one subordinate channel for FB\_MultiAxisCustomJob to work. Master channels are optional.

Subordinate channels and master channels must be set up before the multi-axis custom job is started.

Subordinate channels or master channels cannot be modified or added while the FB\_MultiAxisCustomJob is running.

To remove all previously connected channels from the group, use the method CleanGroup. This method is only available when the function block is not currently executing a job. If this method is called while the group is active, the system detects an error.

## Starting a Multi-Axis Custom Job

After the subordinate channels and master channels are connected, you can initiate the control sequence for the subordinate axes by calling the method Start of the function block FB\_MultiAxisCustomJob.

The Start method requires a reference to your custom calculation function block through the interface IF\_MultiAxisCustomJobCallback. This interface provides the function block FB\_MultiAxisCustomJob with access to your implementation of the following methods:

* Prepare: Executed once during the initialization phase.
* Calculate: Executed cyclically to generate reference values for the movement of the axes.

When FB\_MultiAxisCustomJob is started, the subordinate channels abort their jobs currently being executed on the subordinate axes to transfer control to the multi-axis custom job.

The Start method returns OK after verifying that the required input is available. It does not wait for all subordinate channels to start. Therefore, the inability of a subordinate channel to start does not affect the return value.

Following the verification by the Start method, the system attempts to activate the subordinate channels. The multi-axis custom job proceeds with the start sequence even if one or more subordinate channels are unable to start (for example, because the axes are in the operating states Disabled, Homing, ErrorStop, Stopping). If subordinate channels cannot be started:

* The methods Prepare and Calculate of the user function block are not called.
* Instead, an error is detected (property etResult of FB\_MultiAxisCustomJob).

## Executing a Multi-Axis Custom Job

After the subordinate channels have successfully started, FB\_MultiAxisCustomJob calls the Prepare method in the first RTP cycle. At this point, the values of the master channels have already been calculated for the current cycle. The Prepare method provides access to the master channel values at the moment of job activation, while the subordinate channels reflect the movement values from the RTP cycle immediately preceding the one in which the Prepare method is executed.

This information can be used, for example,  if a subordinate axis already moves when the job starts and you must synchronize the custom algorithm to the current movement state. Otherwise, the Prepare method is only intended for you to initialize your custom algorithm. Custom code in the Prepare method is not required for operation of the multi-axis group.

Immediately after the Prepare method is executed,  FB\_MultiAxisCustomJob begins calling the Calculate  method in every RTP cycle, including the cycle in which Prepare was called.

In the Calculate method, you provide the calculated position, velocity, and acceleration for each subordinate axis. These values are then written to the corresponding subordinate axis using the WriteMovementValues method of the subordinate channel.

FB\_MultiAxisCustomJob sets these values as movement values for this axis for this RTP cycle without further verifications.

If one or all the subordinate axes transition to the operating state ErrorStop during execution, the affected axis stops. FB\_MultiAxisCustomJob continues to call the Calculate method allowing the custom algorithm to monitor the axis state via the etAxisState property and determine whether to continue or stop the remaining axes.

If the WriteMovementValues method is called for an axis in the operating state ErrorStop,  the system detects an error and indicates that the values cannot be written to the axis.

After an axis has left the group due to a detected error, it cannot be re-integrated during the current execution. Even if the detected error on the axis is cleared, control cannot be regained until the FB\_MultiAxisCustomJob is terminated and a new start sequence is initiated.

To stop a subordinate axis, call the method TriggerErrorStop for the corresponding channel. The system continues to execute the Calculate method for the remaining axes.

If a master channel position cannot be updated (xDataValid is FALSE), FB\_MultiAxisCustomJob remains active. Your custom algorithm must handle this condition. Upon restarting the job, master channels are re-initialized.

Writing movement values to a subordinate axis is not possible in the following situations:

|  |  |  |
| --- | --- | --- |
| Operating state subordinate axis | Detected error | Response of other subordinate axes |
| Disabled | AxisIsDisabled | Remain in the operating state SynchronizedMotion |
| ErrorStop | AxisInErrorStop | Remain in the operating state SynchronizedMotion |
| Stopping | JobAborted | Remain in the operating state SynchronizedMotion |
| Standstill | JobAborted | Remain in the operating state SynchronizedMotion |

If a subordinate axis or a master axis performs a motion job before the multi-axis custom job is started, that motion job is aborted.

Homing in conjunction with FB\_MultiAxisCustomJob:

| Type of homing, condition | Axis type | Response |
| --- | --- | --- |
| MC\_Home | Master | FB\_MultiAxisCustomJob continues in the state Executing.  MC\_Home detects an error  (AxisIsUsedAsMasterForSynchronousMotion) |
| MC\_Home | Subordinate | If FB\_MultiAxisCustomJob is started while an axis is homing, the system detects an error (JobStartedWhileAxisIsHoming). |
| FB\_HomingDigitalInput / FB\_HomingTouchProbe | Master | FB\_MultiAxisCustomJob continues in the state Executing.  The operating state of the axis transitions to Homing, no error is detected. |
| Activation of FB\_MultiAxisCustomJob | Master | FB\_MultiAxisCustomJob transitions to the state Executing.  The operating state of the subordinate axes transitions to SynchronizedMotion, but the MasterChannel data is invalid. |

NOTE: Homing of a subordinate axis while FB\_MultiAxisCustomJob is executing is not supported.

## States of a Multi-Axis Custom Job

The state of FB\_MultiAxisCustomJob is indicated through the property etJobState:

| State | Condition |
| --- | --- |
| Idle | FB\_MultiAxisCustomJob is not active. This is the initial state, or the state reached after the methods CleanGroup or ForceStop were successfully called. |
| Linked | The method Start was called. The system is preparing the synchronization of the axes. |
| Executing | The system is in control of the axes. The Calculate method is executed in each RTP cycle. |
| Error | An error has been detected during the start sequence or during execution. |
| Done | The execution has completed successfully because the output q\_xDone of the Calculate method was set to TRUE. |

NOTE: When the function block is in the states Done or Error, you can call the Start method to initiate a new start sequence.

## Terminating a Multi-Axis Custom Job

There are two ways of terminating a started multi-axis custom job:

* The method Calculate of the interface [IF\_MultiAxisCustomJobCallback](IF_MultiAxisCustomJobCallback-Calcu-4D67945D.html) has an output q\_xDone. If this output is set to TRUE, the Calculate method is no longer called. The operating states of the axes transition to Standstill.
* The method ForceStop of the function block FB\_MultiAxisCustomJob terminates the multi-axis custom job. The method Calculate is no longer called. The subordinate axes are decelerated to a standstill with the values of the parameters ErrorStopDec and ErrorStopJerk set for these axes.

## Getting the Position of a Master Channel

If a master channel is connected to the axis group, it provides a position reference for your custom algorithm, based on the movement of the connected master axis.

The starting position of the master channel is the absolute position of the master axis at the moment FB\_MultiAxisCustomJob is started (determined during the RTP cycle prior to the execution of the Prepare method). The system updates the master channel position every cycle before calling the Calculate method. The update is performed by calculating the distance (position delta) the master axis moved during the cycle and adding that distance to the position value of the master channel.

Only the distance covered by the movement of the master axis is considered. Jumps in the absolute position of the master axis caused by modulo rollovers or using  MC\_SetPosition are ignored.

To modify the position value of the master channel without affecting the physical axis, you can call the method SetPosition of the FB\_MasterChannel instance. This method functions similarly to MC\_SetPosition. You can set the master channel position to an absolute or relative value. Following the call, the cyclic update continues to add the movement distance of the axis to the new target position, creating a parallel movement with the defined offset.

If the system cannot determine the distance covered by the movement of the master axis (for example, due to a Sercos communication interruption), the property xDataValid of the master channel is set to FALSE. This indicates that it is no longer possible to follow the master axis. After xDataValid is set to FALSE, it remains FALSE, even if the condition that has caused the inability to determine the correct value is resolved. If the method GetMovementValues is called while xDataValid is FALSE, the system detects an error indicating that the master data is invalid.

## Limitations

Encoder axes (FeedbackAxis) and logical encoder axes (LogicalEncoder) are not supported. The following errors are detected:

* FeedbackAxis: NotSupportedWithFeedbackAxis
* LogicalEncoder: NotSupportedWithLogicalEncoder

The number of master axes is limited to 4.

The number of subordinate axes is limited to 16.
