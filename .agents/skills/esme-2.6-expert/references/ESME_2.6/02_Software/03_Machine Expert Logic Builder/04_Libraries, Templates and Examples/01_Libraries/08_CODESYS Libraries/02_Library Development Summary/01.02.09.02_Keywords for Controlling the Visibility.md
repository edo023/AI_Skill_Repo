## Keywords for Controlling the Visibility

The main difference between controlling visibility with attributes to the method of using keywords is that these keywords will not simply hide symbols. The related symbols are really not accessible.

To encapsulate parts of your application has naturally some pros and cons.

### Pros

* Because the implementation of a function block is not known to other classes, the implementation can be modified without affecting collaboration with other function blocks.
* Increased clarity as only the public interface of a class needs to be considered (Debugging, Documentation).

### Cons

* Unit testing a maximum encapsulated function block is not so easy, because the access to specific internals is not possible.

  **Note**

  With the property [UnitTestingDefine](properties.html#properties_library_properties__properties_unit_testing) there is good way to handle such situations.

| PUBLIC: | This is the default. If no access modifier or the keyword PUBLIC is used, no access limitation is defined. |
| PROTECTED: | Access is only allowed for the own members or members of function blocks which extends the current function block. |
| INTERNAL: | Access is only allowed for the members of the current library. Symbols which are decorated with this keyword are not accessible and not visible from outside of the library. |
| PRIVATE: | Access is only allowed for the own members of a function blocks. |
