# Multicore

## Overview

This function is exclusive to Multicore controllers. Consult the *Programming Guide* specific to your controller for further information about whether the controller supports multicore.

If an EcoStruxure Machine Expert application with multiple tasks is executed on a multicore controller without using the multicore feature, the distribution of tasks is automatically performed by the scheduler of the operating system and no manual intervention is possible.

The multicore feature allows you to assign IEC tasks to dedicated processor cores, taking the acquired number of processor cores into account. This can result in improved performance. As a prerequisite, it is a good practice to split the IEC program into several tasks.

Two different strategies are supported for distributing IEC tasks to different processor cores:

* Each IEC task is linked and fixed to a specific processor core.

  The task is executed on one specific processor core. By grouping tasks, multiple tasks can be linked to the same processor core. This is useful, for example, if the IEC program cannot run on multiple processor cores simultaneously.
* IEC tasks are executed on all available processor cores.

  The operating system distributes one or more tasks of a group to the processor cores.

NOTE: For data consistency without using multicore operators, assign a task group/tasks for one machine (for example, motion, inputs/outputs, HMI) to a single core.

## Considerations Concerning Distributing Tasks over Multiple Processor Cores

Consider the following specific behavior of the IEC program when distributing IEC tasks to different processor cores.

* Task processing according to the priority is not available. The assigned priority is only considered when tasks are grouped to one processor core.
* Cycle consistency of data in the IEC task with the highest priority is not available. To use consistent values within one cycle, data must be copied locally at the beginning of the IEC task cycle.
* A shared process image is provided to all tasks. The (internal) `ReadInputs()` function is called at the start of each task and the `WriteOutputs()` function is called at the end. These functions read from and write to the shared process image. Thus, the Input Processing Output (IPO) model is valid for each individual task.

  When the bus cycle task is called, the shared process image is physically written and the packets are transmitted. The bus cycle task itself waits for each call of `WriteOutputs()` and `ReadInputs()` of the tasks. This helps to ensure data consistency within the bus cycle task. Configure the bus cycle task in the [PLC settings view](D-SE-0083392.html).

  + Outputs can only be assigned to one task. To display the I/O access, open the [Task Deployment view of the device editor](D-SE-0083394.html).
  + Inputs can be assigned to multiple tasks, but direct access is not allowed. Atomic operations are required to write the inputs to local variables of the task.
  + A second task that updates the process image with `ReadInputs()` in parallel to the first task also updates the inputs of the first task while it is running.
  + Data consistency of a task can be ensured only on single cores for the task with the highest priority.

* For consistent counters (incrementer, decrementer), use the atomic external library function SysCpuAtomicAdd() (refer to the SysCpuHandling.library).

## Considerations Concerning Data Consistency

Concerning data consistency, consider the following.

* Bit access (data type BIT) is not processed consistently (atomically) on multicore controllers in the IEC program. It is a good practice to use the external library function SysCpuTestAndSetBit() (refer to the SysCpuHandling.library).
* Simple data types up to a width of 32 bits (such as BOOL, BYTE, WORD/INT, DWORD/DINT) are processed consistently (atomically) in the IEC program on multicore controllers.
* Data types with 64 bits (LINT, LWORD, and LREAL) are processed consistently (atomically) in the IEC program exclusively on 64-bit systems and multicore systems.
* For accessing complex data types (STRING, FB, STRUCT, and ARRAY), use specific measures, such as copying data locally, using a Global Variable List (tasklocal) or using semaphores to synchronize the data and keep consistency.
* In the [Variable Usage tab of the task configuration](D-SE-0088047.html), you can verify whether a variable in an IEC task has read or write access.
* On multicore systems, a memory reordering effect can occur. You can use the IEC operator `__MemoryBarrier()`.

## How to Distribute Tasks Over Multiple Processor Cores

When at least two tasks are defined in your application, for example, `MainTask` (IEC task) and `LowTask` (IEC task) proceed as follows to distribute them over multiple processor cores.

| Step | Action |
| --- | --- |
| 1 | Open the Task Configuration editor. |
| 2 | Open the [Task Groups tab](TaskGroupsTab-CE1329A2.html).  **Result**: The default IEC-Tasks group is displayed. |
| 3 | Click the Add Group button.  **Result**: A new task group with the default name `NewGroup` is created. |
| 4 | Double-click the `NewGroup` field and change the name to `LowGroup`. |
| 5 | From the Core field list, select an assignment, for example, 1. |
| 6 | Open the task `LowTask` (IEC task) in the editor. |
| 7 | From the Task group list, select the new group `LowGroup`.  **Result**: The `LowTask` task is processed by processor Core = 1 as it is a member of group `LowGroup`. |
