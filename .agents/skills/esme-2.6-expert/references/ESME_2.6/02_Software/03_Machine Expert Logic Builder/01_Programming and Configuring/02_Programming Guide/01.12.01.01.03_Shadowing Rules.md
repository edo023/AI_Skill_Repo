# Shadowing Rules

## Overview

In general, there are no restrictions in EcoStruxure Machine Expert that prevent you from using the same identifier for different elements. You can, for example, assign the same name to a local function block instance and to a function.

In the following “bad practice” example, it is unclear whether the instance or the function is called in the program:

```
FUNCTION YYY : INT
;
END_FUNCTION

FUNCTION_BLOCK XXX
;
END_FUNCTION_BLOCK

PROGRAM PLC_PRG
VAR
    YYY : XXX;
END_VAR
YYY();
END_PROGRAM
```

To help prevent creating ambiguous code, make frequent use of the following good practices:

* Follow naming conventions, for example, by using certain prefixes for variables. For further information, refer to the [*Best Practices for the Naming of Identifiers*](D-SE-0083600.html#D-SE-0083600).
* Use the Code Analysis component provided with EcoStruxure Machine Expert for verifying the naming of variables.
* Make consistent use of the qualified\_only [attribute](D-SE-0083651.html#D-SE-0083651) for enumerations and global variable lists.
* Use the library [namespace prefix](D-SE-0083686.html#D-SE-0083686) to access elements of the library.
* Use the prefix `__POOL` to explicitly call a POU from the Global node of the Applications tree and to avoid calling a POU of the same name from the Devices tree.

  Example: `svar_pou := __POOL.POU();`

## Compiler Shadowing Multiple Declarations

The EcoStruxure Machine Expert compiler does not report errors if the same identifier is used for different elements but it searches the code for the declaration of the identifier in a specific order. As soon as a declaration is found, the search is stopped. Other declarations that might exist in the code cannot be detected, they are “shadowed” for the compiler. The following paragraph describes the shadowing rules of the compiler, i.e. the order that is applied when searching for declarations of identifiers. The paragraph [*Ambiguous access and qualified access*](#D-SE-0106992__D-SE-0106992.6) provides ways to help prevent ambiguous access and to bypass shadowing rules.

## Search Order in the Application

When the compiler detects a single identifier in the code of an application, it searches for the corresponding declaration in the following order:

| Stage | Substage | Description |
| --- | --- | --- |
| 1 |  | Local variables |
|  | a | Local variables of a method. |
|  | b | Local variables in the function block, program, or function, and in base function blocks. |
|  | c | Local methods of the POU. |
| 2 |  | Global variables |
|  | a | Global variables in the application if the `qualified_only` attribute is not set in the variable list where the global variables are declared. |
|  | b | Global variables in a parent application if the `qualified_only` attribute is not set in the variable list where the global variables are declared. |
|  | c | Global variables in referenced libraries when neither the library nor the variable list requires qualified access. |
| 3 |  | POU or type names |
|  | a | POU or type names from the application (such as names of global variable lists, function blocks). |
|  | b | POU or type names from a parent application. |
|  | c | POU or type names from a library. |
| 4 |  | Libraries |
|  | a | Namespaces of locally referenced libraries and libraries that are published by libraries. |
| 5 |  | Global variables in the Applications tree |
|  | a | Global variables in the Applications tree if the `qualified_only` attribute is not set in the variable list where they are declared. |
|  | b | POU or type names from the Applications tree (such as names of global variable lists, function blocks). |
|  | c | Libraries from the Applications tree. |

NOTE: Libraries that are inserted in the Library Manager that is a subnode of the Global node of the Tools tree are available in all applications in the project with the placeholder resolution. These libraries form a common namespace with the libraries in the application.

As described in the section [Description of the Tools Tree](D-SE-0083356.html#D-SE-0083356__D-SE-0083356.9), the Library Manager is available at two different locations in the Tools tree: as subnode of the Global node and as subnode of the Application node.

NOTE: In case the same library was added to both Library Manager nodes, the symbols of the library below the Application node are used.

## Search Order in the Library

When the compiler detects a single identifier in the code of a library, it searches for the corresponding declaration in the following order:

| Stage | Description |
| --- | --- |
| 1 | Local variables of a method. |
| 2 | Local variables in the function block, program, or function, and in base function blocks. |
| 3 | Local methods of the POU. |
| 4 | Global variables in the local library if the qualified\_only attribute is not set in the variable list where the global variables are declared. |
| 5 | Global variables in referenced libraries when neither the library nor the variable list requires qualified access. |
| 6 | POU or type names from the local library (such as names of global variable lists, function blocks). |
| 7 | POU or type names from a referenced library. |
| 8 | Namespaces of locally referenced libraries and libraries that are published by locally referenced libraries. |

## Ambiguous Access and Qualified Access

These search orders cannot completely eliminate ambiguous access. Ambiguous access still exists, for example, if a variable with the same name exists in two global variable lists that do not require qualified access. If such a case is detected, the compiler reports the error: `ambiguous use of the name XXX`. To help avoid this, use qualified access, for example by adding the name of the global variable list (example: `GVL.XXX`).

Use qualified access to help avoid shadowing:

* Use the name of the global variable list to provide unique access to a variable in the list.
* Use the name of a library to provide unique access to elements in the library.
* Use the THIS pointer to provide unique access to variables in a function block even if a local variable with the same name exists in a method of the function block.

To find the declaration location of an identifier, execute the Edit > Browse > Go to Definition [command](../../../../../api/crossBook?lang=en-US&virtualBookName=SoMMenu&topicID=D_SE_0083907).

## Searching in Instance Paths

The search orders described in the previous paragraphs do not apply to:

* Identifiers that exist as components of an instance path.
* Identifiers that are used as inputs in calls.

When attempting to access `yy.component`, the entity `yy` determines the location that is searched for detecting the declaration of `component`.

| If... | Then ... |
| --- | --- |
| If `yy` denotes a variable with a structured data type (STRUCT or UNION), | Then `component` is searched in the following order:   1. Local variables of the function block 2. Local variables of the base function block 3. Methods of the function block 4. Methods of the base function block |
| If `yy` denotes a global variable list or a program, | Then `component` is searched in this list only. |
| If `yy` denotes a namespace of a library, | Then `component` is searched in this library using the order as described in the paragraph [*Search Order in the Library*](#D-SE-0106992__D-SE-0106992.5). |

In the second instance, the compiler evaluates whether access to the element is allowed (the variable may only be locally accessible, or a method may be private). If access is not allowed, an error message is generated.
