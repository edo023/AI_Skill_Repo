# Building Applications

## Overview

You can perform an offline code generation in order to verify whether compilation errors are detected before downloading the code to the device. For a successful login, the code generation must have been completed without detecting errors.

## Code Generation, Compile Information

Machine code is generated in the following cases:

* When the [Application](D-SE-0083435.html#D-SE-0083435) project is downloaded to the target device (controller, simulation target).
* When the Build > Generate code command is executed.

At each download, the compile information, containing the code and a reference ID of the loaded application, is stored in the project directory in a file *<projectname>.<devicename>.<application ID>.compileinfo*. The *compileinfo* file is deleted when the Clean or Clean all command is executed.

The build process verifies the project in order to detect programming errors. Detected programming errors are displayed in the Messages view (message category Build).

## Messages at Code Generation

At each code generation, information about memory allocation is displayed in the Messages > Build view.

| Memory allocation information | | Description |
| --- | --- | --- |
| Memory area <n> | | The contents of the reserved memory areas are displayed:  The memory area in which the various types of data and the code are stored depends on the controller.  NOTE: The code and data area consumption is contained in a message similar to:   ```   [INFORMATION]   Memory area 4 contains  Data, Code and Nonsafe data: size: 10436608 bytes, highest used address: 107752, largest contiguous memory gap: 10328856 bytes (98 %) ```   For the addresses `%I`, `%M`, `%Q`, memory is allocated, even if no variable is assigned to an address. In the Messages view, the memory areas for `%M`, `%I`, `%Q` are indicated as Memory, Input and Output.  When you run a Build > Clean command, the memory is reallocated and defragmented.  Memory fragmentation (gaps in the memory) is caused by consecutive incremental builds and online changes. This leads to the following effects:   * The number of memory gaps increases. * The largest contiguous memory gap decreases. |
|  | Highest used address (Byte) | The highest reserved address in the allocated memory area is displayed. During the first build after a cleaning operation, the memory addresses are assigned to variables in ascending order, taking into consideration the natural alignment of the controller (usually 8 bytes). As a result, the highest address used at this time corresponds approximately to the amount of memory used. The rest of the allocated memory area is at this time available for incremental builds and online changes. |
|  | Largest contiguous memory gap (in bytes) | The memory size that is effectively available is displayed in bytes and as percentage of the total memory.  Gaps that were created in the allocated memory are reused whenever possible when changes are made.  When, for example, a global variable of type Byte is added, it is placed in the first free byte of the memory. Other elements, such as a function block instance, a variable of the type structure or array, or the code for a POU, need to be stored contiguously and occupy more memory. Thus, they cannot fill such small gaps as the variable of type Byte but must be allocated to the largest contiguous memory gap area.  NOTE: Also refer to the [Online Change Reserve Memory Settings command](D-SE-0087951.html) for how to avoid memory re-allocation of function block instances in case of small modifications on a function block type. |
